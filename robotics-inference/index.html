Robotics Cloud-Inference — Market Map

May 2026 · Should someone build a "fal for robotics"? A go / no-go map of where robot AI runs and whether an independent company can sell it.

🔗 nodes, terms, and companies link to source. Read §1–§3 for the answer; §4–§6 are the reasoning; §7–§9 are reference.

1 · Start here

This map answers one question: should someone build a cloud service that runs the AI "thinking" for robots — and can an independent company actually make money selling it?

Every robot runs AI in three places. Fast control (reflexes, balance) always runs on the robot itself — it can't wait for a network. Training the AI always runs in the cloud. In between sits the thinking layer (planning, perception, deciding what to do next) — the one piece that can run either on the robot or in the cloud. This whole map is about that middle layer.

Bottom line: the cloud demand is real and broad, but a defensible independent business is narrow — about 1 in 5 cases. The likely winners own both the AI and the robots.

2 · The three layers — read this first

Nothing below makes sense until you can tell these three apart. Only the middle one is up for grabs.
LayerWhere it runsWhat it does / why it runs there
Fast control (reflexes, balance · 50–1000 Hz)On the robot — alwaysCloses the loop in under 50 ms with no buffer. A dropped packet at 200 Hz is a fall or a collision, so it can never depend on a network.
Thinking layer (planning, perception, next move · 1–10 Hz)On the robot or in the cloud ← this whole mapTolerates an extra ~200–300 ms by deciding several moves at once (action-chunking), so it's the one piece that can move off the robot. Training (learning from fleet data)In the cloud — alwaysOffline, compute-heavy, no real-time constraint. Already a clear cloud workload; not the contested ground.

3 · Bottom-line verdict & leader watchlist

Verdict. The cloud opportunity is real and broad — across most segments the thinking layer and training both pull work off the robot. But an independent, defensible business selling that serving is narrow: only ~1 in 5 cloud-viable placements reach a "buy a VLA-serving vendor with a moat" outcome (Tree B, §5). The rest is captured by model labs (bundled), built in-house once a fleet scales, done for free on open-source, or absorbed into full robots-as-a-service. The companies positioned to win are the vertically-integrated model + fleet owners (Physical Intelligence, Skild, Figure, Sereact). Today every KILL signal below is true and no GO signal has appeared.
Two KILL signals, explained in plain English:
  • "Value routes to chip + open-source + model lab": the margin pools elsewhere — NVIDIA monetizes silicon and gives serving away to sell it; open-source (LeRobot/openpi) makes the software $0; the model lab monetizes weights + data flywheel and bundles or open-sources the serving tricks. A serving middleman is sandwiched at commodity margins → absorbed.
  • "The right move (be the model + fleet lab) is $1B+ gated": the defensible play (own the model + data flywheel + fleet) needs frontier training compute + hundreds of millions in teleoperation data (~$118–340/hr) + fleet ownership + scarce talent + years of pre-revenue burn. Comps: PI ~$11B, Skild ~$14B, Figure ~$39B, FieldAI $405M, Rhoda $450M — the entry ticket is a $0.4–1.4B round, not a serving seed.

ⓐ BX · core-vs-context test — serving is CORE (build it) if any row on the left holds; it's CONTEXT (source it → BE) only if every row on the right holds. (core-vs-context — a framework from Geoffrey Moore, the strategist who wrote Crossing the Chasm & Dealing with Darwin: "core" = what customers pay you for and rivals can't copy; "context" = everything else, which you should source rather than build)
Serving is CORE → build in-house (any one)Serving is CONTEXT → source it (all of these)
Measurable competitive weapon — your latency / throughput / $-per-inference beats rivals and customers feel it (higher pick rate, lower $/pick, more uptime)Customers neither see nor pay for it — undifferentiated plumbing
Embeds proprietary IP — custom kernels, distillation, action-head runtime, model-specific scheduler that competitors can't copyAn off-the-shelf runtime (vLLM / TensorRT / NIM / openpi) already hits your control rate
Safety / certification-critical and you must own the evidence chain + carry the liabilityNo proprietary IP is lost by outsourcing it
Co-designed with your model — runtime and weights can't be cleanly separatedCleanly separable from your model (stable API boundary)
Owning it changes your unit economics at your scale (already at graduation scale, §9 M3)Below graduation scale — owning it changes nothing yet
ⓑ BE · build-vs-buy economics — BUY (managed / license) wins if any row on the left holds; BUILD (self-operate) wins only if every row on the right holds. (build-vs-rent)
BUY cheaper → B5 (any one)BUILD / self-operate cheaper → L6 (all of these)
Utilization below the crossover — under ~50–60% sustained GPU util / under ~6 GPU-hr/day per accelerator (idle reserved capacity is wasted; managed serverless scales to zero)Sustained high utilization ≥ ~50–60% / over ~6 GPU-hr/day → owned or reserved hardware beats on-demand (break-even ~13–18 mo)
Sub-scale fleet — too few robots / inferences to amortize a dedicated cluster + an ML-infra team (~$0.5–1.5M/yr loaded for 2–4 engineers)Large, stable, homogeneous fleet at graduation scale (100s–1,000s units, §9 M3) to amortize cluster + team
Speed — you need it in weeks, not the 6–12 mo to build, harden, and put a serving stack on-callStable requirements — quarterly (not weekly) retrains, fixed model family (T7 / T12)
Volatile requirements — model family / size still churning; premature to optimize a fixed in-house stackResidency / IP / latency already force self-host (overlaps B5) — marginal cost of also operating is low
Not core (decided at BX) and engineering time has higher-value uses; preserve runway (opex over capex)All-in build TCO (hire + build + maintain + infra) < all-in buy (license / managed × fleet × duration)
Cross-links: at scaling (§9), BE tips to build / self-operate and B5 to self-host; the A2.5 private-5G-vs-onboard-silicon capex trade feeds BE; a fleet-security finding (trigger T10, §9) routes placement onboard, secondarily shifting capture in-house. Net: only ~1 in 5 cloud-viable placements reach a genuine independent-vendor leaf (L4b + L5); the rest go bundled / build-or-self-operate-OSS / commodity-cloud / RaaS.
