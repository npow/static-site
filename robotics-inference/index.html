Robotics Cloud-Inference — Market Map

May 2026 · evaluated per inference layer, split into a Placement decision and a Capture (business) decision

🔗 Nodes, leaves, terms, and companies link to source. Key correction: a robot is not one verdict — it runs several inference layers (fast reflex · thinking layer · training), each placed separately; only some of those placements support an independent business.

0 · Read this first — evaluate per layer, then ask two separate questions

  • Fast control — the reflex loop (balance, force-contact, collision-stop; 50–1000 Hz): always ONBOARD, plus a fail-safe if the link drops. Not a cloud business.
  • Training & fleet learning (offline: training, RL, fleet routing, sim, analytics): always CLOUD — no real-time constraint.
  • The thinking layer (planning, perception, the vision-language-action model deciding what to do; ~1–10 Hz, tolerates a short delay): this is the layer the decision is actually about → §1 below.
§1 (where it runs) places the thinking layer. §2 (the business question) — only when that layer runs off the robot — asks whether an independent company can charge for it. A robot ends up with a verdict per layer (e.g. a humanoid: fast control onboard · thinking layer hybrid/edge · training in the cloud).

Net: of cloud-viable, value-positive thinking-layer placements, only ~1 in 5 or fewer reach a genuine independent-vendor leaf (L4b + L5) — the rest peel off to the model lab (bundled), to core-build-it-yourself, to $0 free-OSS, to a commodity GPU cloud, to a RaaS integrator, or get out-built by a team the customer hires with the would-be buy budget. And L4b/L5 still must clear the moat gate or collapse to no-defensible-business (the OctoML/Deci/CentML acqui-absorption pattern).

4 · Decision-criteria glossary (quantified, with evidence)

TermQuantified definitionSource
The three inference layersFast control (reflex) = 50–1000 Hz, sub-50 ms, no buffer → onboard. Thinking layer (planning/perception) = 1–10 Hz, tolerates +200–300 ms via action-chunking → cloud-eligible. Training/fleet-learning = offline, no live dependency → cloud.RTC
Connectivity (A3)4 dims, not one. Reliable = p99 RTT ≤ ~50 ms · jitter ≤ ~20 ms · uplink ≥ ~10 Mbps (3-cam compressed stream ≈ 2–10 Mbps) · loss <0.1% → wired / private-5G (URLLC sub-1 ms, 99.999%, µs jitter). Intermittent = median OK, p99 fails (public LTE/5G; contended WiFi6 is non-deterministic). None = ≤4G (RTT 30–70 ms + variable). Per-location, not per-robot.URLLC · VLA-Perf
Compute fit (A5)Model memory ≈ params × 2 B (FP16) or × 0.5 B (INT4) — a 3B VLA ≈ 6 GB FP16 / 1.5 GB INT4; 7B ≈ 14 / 3.5 GB. Full-onboard = FP16 ≤ onboard VRAM (Orin 32–64 GB / Thor 128 GB) AND throughput ≥ control rate (Thor ~19 Hz on a 3B VLA, H100 ~162 Hz) within 40–130 W. Quantized-onboard = fits only after INT4/INT8. Offload = even 4-bit < control rate on the onboard accelerator within power (e.g. >~7–10B at >30 Hz).Thor · VLA-Perf
Placement tier (A6)onboard → on-prem-edge (sub-ms, no WAN, egress≈0) → near-edge/MEC (~10–14 ms) → far-cloud (egress + sovereignty). "Cloud" is not monolithic.—
Value beats costFleet-lifetime TCO: cloud (GPU $/hr ÷ tasks/hr, net of batching/compression/discounts) + egress (~$0.09/GB) vs onboard amortized capex+energy+maintenance. H100 ~$1.4–8/hr; fires only on the far-cloud tier (edge egress≈0). Re-eval as prices fall.egress · GPU $/hr
Build vs buy (BX core-vs-context · BE economics)Not "do they have a team" — a team is buyable with the buy budget. BX: is serving a CORE differentiator (build it) or commodity context (buy/rent)? BE: at expected scale × duration, all-in BUILD cost (hire + build + maintain + keep-pace-with-OSS) vs all-in BUY cost (fees + integration + switching + margin). Build wins at large, sustained scale (the build-vs-rent crossover: cloud wins <~70% util, on-prem only at 80%+; real teams run 40–65%) — that's "graduation"; buy wins sub-scale or for speed-to-market.build-vs-rent
Captive owner funds a bridge (TELEOP)When off-robot is foreclosed AND the robot can't carry a GPU: a deployment owner pays for a human teleoperator (~$22–120/hr, 1:1–1:50) or compute subsidy as a stopgap. Teleop-as-a-service is itself an independent business.teleop econ

5 · Caveats — noted, not branched (to avoid over-engineering)

  • Dynamic economics: GPU $/hr falls, distillation shrinks models → onboard becomes viable over time. Date-stamp the price assumption; treat each leaf as (verdict, re-eval-trigger).
  • Graduation churn: successful 3rd-party-serving customers internalize at scale — the serving TAM is the pre-scale inflow, not the installed base. TELEOP/HYBRID are explicitly transitional.
  • Security / fleet-correlated blast radius: a compromised cloud endpoint can command a whole homogeneous fleet → requires signed commands + per-robot authz + onboard kill-switch on any cloud-serving leaf.
  • HYBRID needs a certified failover: every HYBRID/cloud leaf assumes a certified onboard safe-fallback; with none, "intermittent" collapses to onboard-or-nothing.
  • Connectivity as capex: fixed-site fleets can install private-5G (engineerable); mobile/outdoor/field robots take connectivity as given.
  • Model family: chunkability (A4) and quantizability (A5) depend on architecture (flow-matching/diffusion chunk well; autoregressive differently).
